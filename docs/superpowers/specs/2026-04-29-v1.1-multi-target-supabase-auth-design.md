# V1.1 — Multi-Target + Supabase JSON Auth — Design Spec

**Status:** Draft, pending user review
**Date:** 2026-04-29
**Owner:** Erik (`elagerway`) / Snapsonic
**Forcing function:** Make `casa-ready` actually scan Magpipe (Supabase Auth + Supabase Edge Functions)

## Problem

V1 (`v0.1.0`) ships a working ZAP pre-scan kit, but it can't scan Magpipe in any meaningful way:

1. **Auth doesn't work.** Magpipe uses Supabase Auth (`POST /auth/v1/token?grant_type=password` with JSON body, returns a JWT). V1 only supports ZAP's form-auth (`type=2`), which sends URL-encoded form bodies — Supabase rejects that as `415 Unsupported Media Type`. ZAP falls back to anonymous crawl, missing all authenticated surfaces.

2. **Coverage is too narrow.** V1 scans one URL. Magpipe's API surface lives at `https://<project>.supabase.co/functions/v1/*` — a different origin from the SPA at `magpipe.ai`. V1 has no way to scan both in one invocation; the user would have to run twice with different configs and produce two unrelated TAC submission bundles.

V1.1 fixes both gaps so an actual `casa-ready scan` against Magpipe staging produces meaningful, comprehensive output ready for the TAC portal.

## Goals (V1.1)

1. New auth type `supabase-jwt` that performs Supabase JSON-body login, extracts the access token, and includes it in subsequent requests as `Authorization: Bearer <JWT>` plus `apikey: <anon-key>`.
2. Periodic JWT refresh via ZAP's `pollurl`/`pollfreq` mechanism so long scans don't fail mid-run when the 1-hour token expires.
3. Multi-target scanning: one config can describe multiple targets (e.g., `spa` + `api`), each with its own URL and auth, scanned sequentially in one `casa-ready scan` invocation.
4. Per-target output dirs under the timestamp dir; aggregated top-level `summary.md`.
5. Best-effort per-target failure handling: a failed target collects its error and reports it in the top-level summary; remaining targets still scan.
6. Backward incompatible config change is OK (we're our own only V1 user). New tag will be `v0.2.0`.

## Non-goals (V1.1)

- Generic `json-script` auth that handles arbitrary JSON-API providers (Auth0, Firebase, etc.) — deferred to V1.2 if a second use case emerges.
- Plugin architecture for auth providers — deferred to V1.3+.
- `ScanEngine` class refactor — deferred to V2.
- Per-target scan flavors (e.g., `casa` for SPA + `baseline` for API) — single `--scan` flag still applies to all targets in this release.
- Per-target failure modes (`required: true | false`) — every failure is non-fatal in the loop, but presence-of-any-failure causes non-zero exit.
- Real ADA-tuned ZAP policy file (still using the OWASP Top 10 fallback from V1).
- OAuth-flow scanning (Gmail-restricted user paths) — that's V2.

## Decisions log

| # | Question | Decision | Rationale |
|---|---|---|---|
| 1 | Multi-target shape | **Single config, `targets: [...]` array per env** | Maps to reality (Magpipe = SPA + API); composes with auth type variation; right primitive for V2 OAuth target |
| 2 | Auth abstraction | **Supabase-specific `type: 'supabase-jwt'` for V1.1** | YAGNI — only one JSON-API use case today. Generic `json-script` is V1.2 when a second user shows up |
| 3 | Output structure | **`scan-output/<env>/<ts>/<target>/` subdirs + top-level aggregated `summary.md`** | Clean for TAC bundle (zip the timestamp dir); one summary tells the human story |
| 4 | Failure semantics | **Best-effort: run all targets, collect failures, exit non-zero if any** | Matches "don't lose half a scan to one timeout" |
| 5 | Scan flavor per target | **Single `--scan` flag applies to all targets** | Simple. Per-target flavor deferred to V1.2 |
| 6 | JWT refresh | **Wire ZAP's `pollurl`/`pollfreq`/`pollunits` to re-auth every 50 min** | Native ZAP support for type=4. Cheap to configure |
| 7 | CLI no-`--target` behavior | **Run all targets sequentially** | "One command for the full scan" |

Plus implementation approach: **Approach 1 (minimal additive)** — add `auth/` module with one file per type, refactor `runScan` to loop. Avoids premature plugin abstraction (Approach 2) and bigger ScanEngine refactor (Approach 3); both can land later when justified.

## Config schema (V1.1)

```javascript
export default {
  app: 'magpipe',

  envs: {
    staging: {
      targets: [
        {
          name: 'spa',
          url: 'https://staging.magpipe.ai',
          auth: {
            type: 'form',
            loginUrl: 'https://staging.magpipe.ai/login',
            loginRequestBody: 'email={%username%}&password={%password%}',
            usernameField: 'email',
            passwordField: 'password',
            loggedInIndicator: 'Sign out|/dashboard',
          },
        },
        {
          name: 'api',
          url: 'https://x.supabase.co/functions/v1',
          auth: {
            type: 'supabase-jwt',
            loginUrl: 'https://x.supabase.co/auth/v1/token?grant_type=password',
            apiKey: process.env.SUPABASE_ANON_KEY,
            refreshSeconds: 3300,  // 55 min, comfortably under 1-hour JWT expiry
          },
        },
      ],
    },
    prod: {
      targets: [/* same shape, prod URLs */],
    },
  },
};
```

**Validation rules:**
- `app`: non-empty string (unchanged)
- `envs[name].targets`: non-empty array
- Each `target`: `name` (non-empty string, unique within env), `url` (non-empty string starting with `http`), `auth` (object)
- `auth.type`: must be `'form'` or `'supabase-jwt'` (validated against the dispatcher's known list)
- For `auth.type: 'form'`: same 6 fields as V1
- For `auth.type: 'supabase-jwt'`: `loginUrl`, `apiKey` (non-empty), `refreshSeconds` (positive integer, optional, default 3300)

**Credentials:** still env-var-only (`CASA_READY_USER`, `CASA_READY_PASS`). The `apiKey` for Supabase is the public anon key — safe in config but supported via env (`process.env.SUPABASE_ANON_KEY`) for parity.

## Components

| Path | Purpose |
|---|---|
| `cli/lib/auth/index.js` | Exports `getAuthContext(authConfig, values, runId)` — dispatches to the right module by `authConfig.type`, returns `{ contextXml, scriptPath?, contextPath }` for the orchestrator to mount |
| `cli/lib/auth/form.js` | Form-auth context generation — extracts V1's logic from `scan.js`. Reads `form-context-template.xml`, renders, returns rendered XML |
| `cli/lib/auth/supabase-jwt.js` | Supabase JWT auth — reads `supabase-jwt-context-template.xml` AND copies/renders `supabase-jwt-script.js` to a tmp path. Returns context XML + script path |
| `cli/lib/targets-summary.js` | Reads each target's `summary.md`, produces a top-level `summary.md` with sections per target plus a "Failed targets" section if any |
| `configs/zap/form-context-template.xml` | Identical to V1's `context-template.xml`, just renamed |
| `configs/zap/supabase-jwt-context-template.xml` | ZAP type=4 (script-based) auth context. References the script by name; configures `pollurl` for refresh |
| `configs/zap/supabase-jwt-script.js` | Vendored JS that runs inside ZAP's Nashorn/Graal interpreter. Performs Supabase login, parses JWT from response, sets it in a token variable ZAP injects into subsequent requests |

## Data flow

1. CLI runs `casa-ready scan` (or with `--target spa` to filter to one).
2. `config.js` loads + validates `casa-ready.config.js`. Returns `{ app, envs }`.
3. `scan.js` resolves env to `targets[]`. If `--target <name>` provided, filters to that one (errors if not found).
4. Initialize `failures = []`, `targetSummaries = []`.
5. **For each target:**
   1. `auth/index.js` dispatches by `target.auth.type` → returns `{ contextXml, scriptPath? }`
   2. `writeContext` writes the context XML to a tmp path
   3. If `scriptPath` returned, `writeScript` copies the auth script to tmp (separate dep so test-mockable)
   4. `mkdirOutput(env, ts, target.name)` creates `scan-output/<env>/<ts>/<target>/`
   5. `buildZapArgs` (extended) includes the script mount if present
   6. `runZap` executes
   7. `readResultsJson` + `summarize` produce per-target `summary.md` and `results.txt`
   8. Wrapped in try/catch: failure → `failures.push({ target, error, stage })`; finally → cleanup all tmp files for this target
6. After loop: `targets-summary.js` reads all per-target summaries + failures list, writes top-level `summary.md` and `results.txt`.
7. Exit code: `0` if `failures.length === 0`, else `1`.

## Error handling

| Failure mode | Behavior |
|---|---|
| Config has no `targets[]` for the env | Validate-time error: "env staging has no targets — see casa-ready.config.example.js" |
| Two targets in same env share a `name` | Validate-time error: "duplicate target name 'spa' in env staging" |
| Unknown `auth.type` | Validate-time error: "unknown auth.type 'bogus' — must be one of: form, supabase-jwt" |
| `--target <name>` doesn't match any configured target | Exit 1 before any I/O: "target 'xyz' not found. Configured targets: spa, api" |
| Supabase JWT login fails (4xx/5xx, malformed response) | ZAP's auth script throws; ZAP marks the user as unauthenticated; the scan continues anonymously. Logged in per-target `summary.md` as "auth failed during scan." Still produces artifacts. |
| `runZap` rejects for one target | Caught; `failures.push(...)`; cleanup runs; loop continues to next target |
| All targets fail | Top-level summary shows the failure list; exit 1 |
| Cleanup itself fails (rm of tmp file) | Logged to stderr but doesn't propagate — the scan is what matters |

**Production-side details:**
- Tmp file naming uses `${target.name}-${Date.now()}` to avoid collision across targets in the same scan
- Each target's cleanup happens in its own `finally` block so a target failure doesn't skip its own cleanup OR the next target's run

## Testing strategy

**Unit (Vitest):**
- `tests/lib/auth/form.test.js` — mirrors V1's `zap-context` tests, asserts the form-context renders correctly through the new dispatcher
- `tests/lib/auth/supabase-jwt.test.js` — asserts the supabase-jwt context XML contains the right `pollurl`, `pollfreq`, script reference, and that `getAuthContext` returns a valid script path
- `tests/lib/auth/index.test.js` — asserts the dispatcher routes correctly + throws on unknown type
- `tests/lib/targets-summary.test.js` — fixtures for 2 successful + 1 failed target, snapshot on the aggregated markdown
- `tests/lib/config.test.js` — extended for the new schema (multi-target, validation rules above)
- `tests/commands/scan.test.js` — refactored: tests multi-target loop, per-target failure-collection, `--target` filter, exit code

**Integration (opt-in):**
- Existing juice-shop smoke test: now uses 2 targets — one form-auth against juice-shop, one fake-baseline against `httpbin.org` (any always-up endpoint). Proves the loop works end-to-end against real ZAP.

**Manual verification (pre-tag):**
- `casa-ready scan --env staging` against magpipe.ai staging. Confirm both `spa/` and `api/` output dirs populate. Top-level `summary.md` shows both targets. JWT refresh fires (visible in ZAP container logs as a re-auth POST every 50 min).

## JWT refresh wiring

Supabase JWTs default to 1-hour expiry. ZAP type=4 supports periodic re-auth via:

```xml
<authentication>
  <type>4</type>
  <strategy>EACH_RESP</strategy>
  <pollurl>{{supabaseUrl}}/auth/v1/user</pollurl>
  <polldata></polldata>
  <pollheaders>Authorization: Bearer {%TOKEN%}\napikey: {{apiKey}}</pollheaders>
  <pollfreq>{{refreshSeconds}}</pollfreq>
  <pollunits>SECONDS</pollunits>
  ...
</authentication>
```

ZAP polls `/auth/v1/user` every `refreshSeconds`. If the response is 401 (token expired), ZAP re-runs the auth script, gets a fresh JWT, and resumes the scan transparently. `refreshSeconds: 3300` (55 min) leaves a comfortable buffer before the 1-hour expiry.

## Open dependencies

1. **Magpipe staging Vercel alias** — same blocker from V1. Until `staging.magpipe.ai` (or a stable Vercel branch alias) exists, the `--env staging` config can't be filled in with real values. Tracked separately.
2. **Supabase anon key** — public, but needs to be in the local `.env.local` and (eventually) the GHA secrets. Add to docs.
3. **Magpipe test user** — same as V1.
4. **JWT-refresh validation** — first scan must run long enough to actually trigger refresh (>55 min). Either we run a slow CASA-flavor scan and verify in container logs, or we drop `refreshSeconds` to 60 in a one-off test.

## Cost-of-being-wrong analysis

| Decision | If wrong, what happens? | Mitigation |
|---|---|---|
| Approach 1 (additive, not plugin) | A 3rd auth type lands and we want a real plugin model | Refactor at that point — current additive shape is a reasonable starting point for that refactor anyway |
| Best-effort failure semantics | A failed target's findings get lost in the aggregate noise | Top-level summary explicitly headers failed targets in their own section |
| Single `--scan` flag for all targets | Want `casa` for SPA + `baseline` for API | V1.2 — small per-target field added to the schema, fully additive |
| Supabase-specific (not generic) JSON auth | Auth0/Firebase user wants to contribute support | They add a new file under `cli/lib/auth/`, contribute upstream. Same pattern. |
| Per-target output subdirs | TAC portal expects flat layout | Easy to add a `--flatten` flag if it turns out wrong; cheaper than the reverse |
