# V1.2 — YAML Config + `init` Command + Schema-Driven DX — Design Spec

**Status:** Draft, pending user review
**Date:** 2026-04-30
**Owner:** Erik (`elagerway`) / Snapsonic
**Forcing function:** OSS launch — make CASA Ready safe and obvious for external customers to adopt

## Problem

V1 + V1.1 + the v0.2.x patch series ship a working tool but a **hostile adoption surface**:

1. **`casa-ready.config.js` triggers security-team alarms.** External customers asking their AppSec teams to add a JS file with `process.env.X` reads + arbitrary-expression capability to their repo invites formal review. Compare to declarative configs (YAML/JSON) that are statically inspectable — what every other security tool in the ecosystem (ZAP, GitHub Actions, Snyk, Dependabot) ships.
2. **No `init` story.** Users copy-paste `casa-ready.config.example.js` and hand-edit. Every other adoption-friendly CLI ships an `init` command (Stripe, Supabase, Prisma, Vercel) — so customers don't author the file at all.
3. **No editor support.** Hand-editing the JS config gets you no autocomplete, no schema validation, no inline docs. External customers won't know the schema and have nothing to lean on.

V1.2 closes all three before we point external customers at the tool.

## Goals (V1.2)

1. **YAML config format** (`casa-ready.yml`) replaces the JS module. Same shape, declarative, statically reviewable.
2. **`${VAR_NAME}` env-var expansion** in YAML so credentials/keys don't get inlined.
3. **`casa-ready init`** — interactive prompts that walk the user through generating a valid `casa-ready.yml`. They never have to read the schema themselves.
4. **JSON Schema** published in the package (`schemas/casa-ready.schema.json`). Users add a one-line `# yaml-language-server: $schema=...` directive at the top of their YAML and get IDE autocomplete + inline validation in VS Code's YAML extension.
5. **TypeScript types** exported (`types/index.d.ts`) for programmatic Node users.
6. **Zod schema as single source of truth** — drives runtime validation, JSON Schema export, and TS type export from one definition.
7. **Hard cutover migration** — v0.3 supports only YAML. JS configs error with a migration guide pointer. (We're our own only V0.2 user.)
8. Ship as **v0.3.0** (breaking config-format change in 0.x = minor bump).

## Non-goals (V1.2)

- **Per-target scan flavor** (`scan: casa | baseline` per-target) — deferred to v0.3.1 if real users ask.
- **Per-target `required: true | false`** — deferred to v0.3.1.
- **Generic `json-script` auth** for Auth0/Firebase/etc. — deferred to v0.4 / when a 2nd JSON-API user shows up.
- **Schema-driven dynamic `init` prompt generation** — `init` walks a hand-coded prompt sequence; the schema validates the result. Reverse direction (prompts derived from Zod introspection) is deferred — too brittle for V1.2.
- **Real ADA-tuned ZAP policy file** — still using the OWASP Top 10 fallback; carries forward from V1.
- **OAuth-flow scanning (V2 OAuth)** — still V2 work.
- **JSON or TOML config alternatives** — YAML only. Adding more formats is a 2nd-user concern.
- **Backward compat with v0.2.x JS configs** — explicit hard cutover.

## Decisions log

| # | Question | Decision | Rationale |
|---|---|---|---|
| 1 | Config format | **YAML** | Universal, comments supported, every adjacent security tool uses it (GitHub Actions, ZAP, Snyk) |
| 2 | Env var expansion syntax | **`${VAR_NAME}`** | POSIX/docker-compose/dotenv standard; lowest friction |
| 3 | `init` UX | **Interactive prompts** (`@inquirer/prompts`) | External customers don't know the schema; walk them through it |
| 4 | Migration | **Hard cutover** | Only V0.2 user is us; deprecation period is dead-weight code |
| 5 | Filename | **`casa-ready.yml`** | Short, discoverable, matches package name (vercel.json pattern) |
| 6 | What else for V1.2 | **Just YAML + init + JSON Schema + TS types** | YAGNI on per-target flags / generic JSON auth — defer to v0.3.1+ |
| 7 | Implementation approach | **Approach 3 — Zod source-of-truth + JSON Schema + TS types** | OSS launch needs polish; schema-first reduces total code once committed; IDE DX is the discriminator vs. competitor tools |

## New YAML config schema

```yaml
# casa-ready.yml
# yaml-language-server: $schema=https://raw.githubusercontent.com/elagerway/casa-ready/main/schemas/casa-ready.schema.json
app: magpipe

envs:
  staging:
    targets:
      - name: spa
        url: https://staging.magpipe.ai
        auth:
          type: form
          loginUrl: https://staging.magpipe.ai/login
          loginRequestBody: 'email={%username%}&password={%password%}'
          usernameField: email
          passwordField: password
          loggedInIndicator: 'Sign out|/dashboard'

      - name: api
        url: https://x.supabase.co/functions/v1
        auth:
          type: supabase-jwt
          loginUrl: https://x.supabase.co/auth/v1/token?grant_type=password
          apiKey: ${SUPABASE_ANON_KEY}
          refreshSeconds: 3300

  prod:
    targets: []  # add prod targets here
```

The `# yaml-language-server: $schema=...` line is the magic that activates IDE autocomplete in VS Code's YAML extension. The `init` command writes this line automatically.

## Components

| Path | Purpose |
|---|---|
| `cli/lib/schema.js` | Zod schemas: `ConfigSchema`, `EnvSchema`, `TargetSchema`, `FormAuthSchema`, `SupabaseJwtAuthSchema`. **Single source of truth** — drives validation, JSON Schema export, TS type export. |
| `cli/lib/config.js` | Refactored: reads `casa-ready.yml`, calls js-yaml parser, hands result to `expandEnv` then to `ConfigSchema.parse()`. Same exports as V1.1 (`loadConfig`, `resolveTargets`, `readAuthCredentials`) — drop-in for the orchestrator. |
| `cli/lib/env-expand.js` | Pure recursive walker. Replaces `${VAR}` in any string node with `process.env.VAR`. Throws on missing var with the dotted path that referenced it (e.g. `envs.staging.targets[1].auth.apiKey`). |
| `cli/commands/init.js` | Interactive prompts via `@inquirer/prompts`. Walks user through app name, env URLs, targets, per-target auth. Each step's output validated against the Zod schema before writing to disk. Writes `casa-ready.yml` + the schema reference comment. |
| `schemas/casa-ready.schema.json` | Auto-generated from Zod via `zod-to-json-schema`. Committed to repo (so the GitHub raw URL is stable for users' `# yaml-language-server: $schema=` directive). |
| `types/index.d.ts` | Auto-generated from Zod schemas. Exports `CasaReadyConfig`, `Env`, `Target`, etc. |
| `scripts/build-schema.js` | Run before publish: regenerates `schemas/casa-ready.schema.json` + `types/index.d.ts`. Wired as `prepublishOnly` script in package.json. |
| `bin/casa-ready.js` | Adds `init` subcommand routing alongside `scan`. |

## Data flow — `casa-ready scan` (config load)

1. Look for `casa-ready.yml` in cwd
2. If absent but `casa-ready.config.js` is present → throw migration error
3. Read YAML file → `js-yaml.load(content)` → parsed object
4. `expandEnv(parsed)` walks the tree, replaces `${VAR}` with env values, throws on missing
5. `ConfigSchema.parse(expanded)` validates structure (Zod throws on shape errors)
6. Returns the validated config (typed as `CasaReadyConfig`)

## Data flow — `casa-ready init`

1. Check for existing `casa-ready.yml` in cwd → if present, prompt to overwrite (default: no)
2. Sequential prompts (`@inquirer/prompts`):
   - **App name** (text, required, validated against `ConfigSchema.shape.app`)
   - **Staging URL** (text, validated as http(s) URL)
   - **Add a prod URL?** (confirm; if yes, prompt for prod URL)
   - For each env, loop "Add a target?" until user says no:
     - Target name (text)
     - Target URL (text, validated)
     - Auth type (select: `form` | `supabase-jwt`)
     - For `form`: loginUrl, loginRequestBody, usernameField, passwordField, loggedInIndicator
     - For `supabase-jwt`: loginUrl, apiKey env var name (default `SUPABASE_ANON_KEY` — emitted as `${SUPABASE_ANON_KEY}`), refreshSeconds (default 3300)
3. Build the config object; validate the WHOLE thing against `ConfigSchema.parse()`
4. `js-yaml.dump()` to YAML string with sensible options (no quoting unless needed, 2-space indent)
5. Prepend the `# yaml-language-server: $schema=...` line
6. Write `casa-ready.yml` to cwd
7. Print: "Done. Next steps: `export CASA_READY_USER=...; export CASA_READY_PASS=...; casa-ready scan`"

## Error handling

| Failure mode | Behavior |
|---|---|
| Missing `casa-ready.yml` | Error: `casa-ready.yml not found in <cwd>. Run \`casa-ready init\` to generate one.` |
| `casa-ready.config.js` present + no `.yml` | Migration error: `Found legacy v0.2.x config (casa-ready.config.js). v0.3 uses YAML. See [link to MIGRATION.md] or run \`casa-ready init\`.` |
| Invalid YAML syntax | Surface js-yaml's line/column error verbatim |
| Missing env var referenced by `${VAR}` | `Missing env var: SUPABASE_ANON_KEY (referenced by envs.staging.targets[1].auth.apiKey in casa-ready.yml)` |
| Schema validation fails | Zod issue list formatted as: `Config validation failed: <path>: <message>` per issue |
| `init` cancelled (Ctrl-C) | Don't write partial file; exit cleanly with no changes to disk |
| `init` overwrite-confirm declined | Print "Aborted. casa-ready.yml unchanged." and exit 0 |

## Migration (v0.2.x → v0.3.0)

Hard break.

1. `loadConfig` checks for `casa-ready.yml` first
2. If absent but `casa-ready.config.js` is present → migration error with link
3. README gets new "Migrating v0.2.x → v0.3.0" section: side-by-side JS-vs-YAML config + the one-line `casa-ready init` shortcut
4. CHANGELOG entry for 0.3.0 names the breaking change explicitly
5. New `MIGRATION.md` at repo root with the full migration guide (referenced by the migration error message)

## Testing strategy

**Unit:**
- `cli/lib/schema.js` — Zod schemas validate correctly-shaped configs, reject mis-shaped ones with useful issues.
- `cli/lib/env-expand.js` — pure-function tests:
  - Single string with one `${VAR}` → expanded
  - Nested object: env vars expanded at any depth
  - Arrays of strings: each element expanded
  - Missing env var throws with the dotted path
  - Literal `$` not in `${...}` form passes through unchanged
  - `${...}` with empty braces or special chars handled (defensive)
- `cli/lib/config.js` — `loadConfig` reads YAML fixture, expands, validates. New error paths: missing yml, legacy js detected, invalid yaml, schema fail.
- `cli/commands/init.js` — mock `@inquirer/prompts` answers; assert generated YAML matches a fixture; assert schema validates the result.
- `scripts/build-schema.js` — runs without error and emits non-empty `.json` + `.d.ts`.

**Integration smoke:**
- `tests/integration/juice-shop.config.yml` (renamed from `.js`); existing smoke logic unchanged.

**Manual verification (pre-tag):**
- `casa-ready init` in an empty dir produces a valid YAML
- The generated YAML's `$schema` reference resolves on GitHub raw
- VS Code with YAML extension installed shows autocomplete on a fresh `casa-ready.yml`
- `casa-ready scan` against juice-shop with the new YAML config matches v0.2.3 behavior

## Open dependencies / questions

1. **JSON Schema URL strategy.** The `# yaml-language-server: $schema=https://raw.githubusercontent.com/elagerway/casa-ready/main/schemas/casa-ready.schema.json` URL points at the `main` branch. Versioning options: tag-based (`/main` → `/v0.3.0`), but tag-based requires users to bump the URL on upgrade. `main` is "always latest schema" which can break older configs. **Default for V1.2: use `main`**. Document the tradeoff.

2. **Zod version.** Zod 3.x is stable, well-tested. Zod 4 is in early access. Pin to `^3.23.0`.

3. **TypeScript types** are emitted but the project itself stays JS. No tsconfig, no compilation. The .d.ts is hand-rolled-via-tooling, lives next to the JS, and gets shipped via `package.json`'s `"types"` field.

## Cost-of-being-wrong analysis

| Decision | If wrong, what happens? | Mitigation |
|---|---|---|
| Approach 3 (Zod + JSON Schema + types) | Significant scope expansion; longer build; more deps to maintain | Each piece is self-contained — drop one if it bites (e.g., skip d.ts if TS export is annoying) |
| Hard cutover migration | A user mid-flight on v0.2.x has to stop and migrate | Only known V0.2 user is us; CHANGELOG + MIGRATION.md cover anyone else |
| Schema URL = `main` not tag | Schema breaks an old config when we land V1.3 | Pin to a tag if it bites; backward-compat changes to schema are rare anyway |
| `init` = hand-coded prompts (not schema-driven) | Two sources of truth (Zod + prompt sequence) drift over time | A single test that exercises `init` end-to-end + asserts result validates against ConfigSchema catches drift |
| Bundle size grows ~150KB | npm install slightly slower; no real cost for a CLI | Acceptable for a tool used episodically (pre-scan + recert) |

## Cut from V1.2 (explicitly deferred)

- Per-target scan flavor (v0.3.1)
- Per-target `required: true|false` (v0.3.1)
- Generic `json-script` auth (v0.4 or when 2nd JSON user ships)
- Schema-driven dynamic init prompts (later)
- TOML or JSON config alternatives (when 2nd user requests)
- Backward compat shim for v0.2.x JS configs (deliberate hard break)
- Real ADA-tuned ZAP policy file (carries forward from V1)
- OAuth-flow scanning (V2)
